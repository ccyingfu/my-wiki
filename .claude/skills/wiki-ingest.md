# Wiki Ingest — 资料录入

> 将新资料编译进 Wiki 的持久知识体系。一次录入，持续复用。

---

## 前置检查

1. **确认路径**：Wiki 根目录 = `/Users/liyingfu/Desktop/my-wiki`
2. **读取规范**：必须先读 `wiki/schema.md` 了解页面模板、命名约定、工作流
3. **读取索引**：读 `wiki/index.md` + 对应子索引 `_index-*.md`，了解现有内容
4. **确认资料存在**：验证 `raw/` 下的目标文件确实存在且可读

---

## 执行流程（7步）

### Step 1：阅读 & 理解原始资料

```
1. 打开并完整阅读用户指定的 raw/ 文件
2. 提取关键信息：
   - 资料类型（研报/文章/公告/政策/数据/笔记）
   - 核心主题和结论
   - 涉及的实体（公司名、板块名、人名、指标名）
   - 关键数据和数字
   - 与现有 Wiki 内容可能的关联点
3. 如果资料较长（>3000字），先输出摘要给用户看：
   "我读了这份《{标题}》，核心内容是：...
    我准备从以下角度录入 Wiki：
    - 创建/更新页面：[列表]
    - 你想重点关注哪些方面？还是按我的判断来？"
```

### Step 2：确定涉及页面清单

```
基于 Step 1 的分析，列出所有需要创建或更新的页面：

新建页面（资料首次引入的新实体/概念）：
  □ wiki/entities/stocks/{股票名}.md        （如果涉及新个股）
  □ wiki/entities/sectors/{板块名}.md       （如果涉及新板块）
  □ wiki/entities/indicators/{指标名}.md     （如果涉及新指标）
  □ wiki/entities/people/{人物名}.md         （如果涉及新人）
  □ wiki/concepts/frameworks/{名称}.md      （如果涉及新框架）
  □ wiki/concepts/themes/{主题}.md          （如果涉及新投资主题）
  □ wiki/concepts/theories/{理论}.md        （如果涉及新理论）
  □ wiki/analyses/reviews/{日期}-摘要.md    （本次录入的分析摘要）

更新页面（已有页面需要补充新信息）：
  □ wiki/entities/stocks/{已有}.md          （补充新数据/观点）
  □ wiki/entities/sectors/{已有}.md         （更新行业动态）
  ...

预计影响：N 个页面（通常 5-15 个）
```

### Step 3：创建/更新页面

**遵循 wiki/schema.md 中的对应模板格式**

每个页面必须包含 YAML frontmatter + 结构化正文。

#### 新建页面的标准结构（以个股为例）：

```markdown
---
title: "{名称}"
type: stock | sector | indicator | person | framework | theme | theory
code: "{代码}"                    # 个股用
sector: "{所属板块}"              # 个股用
tags: [{标签列表}]
created: {今天日期}
updated: {今天日期}
sources: 1                        # 初始=1，后续每+1
status: active | watching | archived
---

# {标题}

## 一句话定位
{一句话说清楚它是什么}

## 核心基本面 / 核心信息 / 定义说明
（根据类型不同，填写对应内容）

## 关键数据 / 要点
（从当前资料中提取）

## 投资逻辑 / 相关性 / 演进
（如有历史信息则记录演进，否则先写当前认知）

## 关联链接
- 所属板块：[[板块名]]
- 相关概念：[[概念名]]
- 相关个股：[[股票名]]

## 资料来源
1. raw/{子目录}/{文件名}
```

#### 更新已有页面时：

```
1. 先读取现有页面完整内容
2. 在对应 section 后面追加新信息（标注来源）
3. 更新 updated 日期为今天
4. sources 计数 +1
5. 如有矛盾 → 在页面内标记 [⚠️ 矛盾] 并同步写入 contradictions.md
6. 如有重要认知变化 → 更新「投资逻辑演进」时间线
```

### Step 4：创建分析摘要页

每次 Ingest 都生成一份摘要页，记录本次录入的核心产出：

```markdown
---
title: "{资料标题} — 录入摘要"
type: ingest-summary
source: "raw/{path}/{filename}"
date: {日期}
pages-created: N
pages-updated: M
---

# {资料标题} — 录入摘要

## 资料概要
- **类型**：研报/文章/公告/...
- **来源**：{机构/作者/发布方}
- **日期**：{原始日期}

## 核心要点
1. ...
2. ...
3. ...

## 录入动作
| 操作 | 页面 | 说明 |
|-----|------|------|
| 新建 | entities/... | ... |
| 更新 | entities/... | 追加了xxx |

## 发现的问题/矛盾
（如有）

## 值得后续关注
（如有）
```

保存到：`wiki/analyses/reviews/{YYYY-MM-DD}-{简短标题}.md`

### Step 5：更新索引

```
1. 更新对应的 _index-*.md 子索引：
   - 新建了个股页 → 更新 entities/_index-stocks.md
   - 新建了板块页 → 更新 entities/_index-sectors.md
   - 新建了概念页 → 更新 concepts/_index-concepts.md
   - 生成了分析页 → 更新 analyses/_index-analyses.md
2. 更新 wiki/index.md 总入口中的计数（页面总数、资料总数）
3. 索引条目格式保持一致
```

### Step 6：更新日志 & 元信息

#### log.md 追加（固定前缀格式）：

```markdown
## [{YYYY-MM-DD}] ingest | {资料简短标题}
- **来源**：raw/{子目录}/{文件名}
- **类型**：{研报/文章/公告/...}
- **涉及页面**：{列出所有创建/更新的页面}
- **关键发现**：{2-3句话总结最重要的信息}
- **矛盾标注**：（如有）→ meta/contradictions.md#C{N}
- **新增问题**：（如有）→ meta/open-questions.md
```

#### 元信息文件更新（按需）：

```markdown
# meta/contradictions.md — 追加新矛盾
| C{N} | 涉及页面 | 矛盾描述 | 来源资料 | 状态 |
|------|---------|---------|---------|------|
| C{N} | A.md vs B.md | ... | 本次ingest | 待验证 |

# meta/open-questions.md — 追加新问题
| Q{N} | 问题 | 关联页面 | 建议 |
|------|------|---------|------|

# meta/reading-list.md — 追加建议后续阅读
（如本次资料提到了值得深挖但未展开的内容）
```

### Step 7：Git 提交

```bash
cd /Users/liyingfu/Desktop/my-wiki
git add -A
git commit -m "ingest: {YYYY-MM-DD} | {资料简短标题}

新建：
- {新建页面列表}

更新：
- {更新页面列表}

涉及 N 个页面，M 行变更"
```

## 批量录入模式

当用户一次提交多份资料时：

```
1. 逐份执行上述流程（Step 1-6）
2. 但在 Step 2 时全局统筹：如果多份资料都涉及同一页面，
   合并为一次更新而不是重复更新
3. 最后统一做一次 Step 7 Git commit
4. 日志中每份资料一条记录，commit message 中汇总列出

Commit message 格式：
"ingest: {YYYY-MM-DD} | 批量录入 (N份资料)
- 资料1: {标题} → 影响 X 个页面
- 资料2: {标题} → 影响 Y 个页面
- ..."
```

**注意**：批量模式下默认减少交互（不逐份询问），除非用户特别指定要逐一确认。

## 特殊情况处理

### 图片/附件类资料

```
1. 图片应已在 raw/assets/ 下
2. 在页面中以 ![[图片名]] 引用（Obsidian 格式）
3. 正文中用文字描述图片的关键信息（因为 LLM 不一定能看到图）
4. 如需让用户确认图片内容，单独提示："请查看 attached image: raw/assets/xxx.png"
```

### 数据文件（CSV/Excel）

```
1. 读取数据文件内容
2. 提取关键统计值（最新值、趋势、异常点）
3. 将结构化数据写入对应的 indicators 页面
4. 原始文件保留在 raw/data/ 不动
5. 可考虑在页面中嵌入精简后的 Markdown 表格
```

### 超长资料（>10000字）

```
1. 分段处理：先读前30%获取整体框架
2. 识别重点章节深入阅读
3. 非核心章节快速扫过提取要点
4. 向用户说明："这是一份很长的资料，我重点阅读了X/Y/Z章节，
   其他章节做了摘要处理。如果你希望我深入某个部分请告诉我。"
```

### 资料与现有内容高度重复

```
1. 检测到重复后告知用户："这份资料的大部分内容已经在 Wiki 中了
   （参见 xxx.md）。新的增量信息是：..."
2. 只录入真正新增的部分
3. 在日志中注明"增量录入"
```

---

## 注意事项

1. **绝不修改 raw/** — 原始资料是只读层
2. **每份资料必有日志条目** — log.md 不能漏
3. **frontmatter 必须完整** — 缺字段 = 后续查询/筛选出问题
4. **链接必须有效** — 创建页面时要同时确保链接目标存在
5. **不确定就问** — 尤其是对矛盾信息的定性判断
6. **不要自作主张修改 schema.md** — Schema 变更需要用户参与决策
