# Wiki Lint — 健康检查与维护

> 让 Wiki 在不断扩张的同时保持健康、一致、可用。

---

## 前置检查

1. **确认路径**：`/Users/liyingfu/Desktop/my-wiki`
2. **读取 Schema**：`wiki/schema.md`（判断合规性）
3. **读取索引**：`wiki/index.md` + 所有 `_index-*.md`（了解全局）

---

## 7项必检

### 🔴 L1 — 矛盾检测

```
方法：
1. 遍历 entities/ + concepts/ 的所有页面
2. 同主题的不同描述交叉比对
3. 重点：
   - 同指标在不同页面的数值差异
   - 同事件的时间线/因果冲突
   - 不同来源对同一公司的评级差异(已标注则跳过)
4. 对照 meta/contradictions.md 已有记录，判断是否仍存在/已解决
```

```markdown
### L1 矛盾检测

| 编号 | 涉及页面 | 矛盾内容 | 可能原因 | 建议 |
|-----|---------|---------|---------|------|
| C1 | stocks/A.md vs ... | PE数值不同 | 数据日期不同 | 统一最新 |
| C2 (已有) | ... | ... | ✅ 已解决 | 可归档 |

→ 新矛盾追加到 meta/contradictions.md
→ 已解决的标记 [已解决]
```

---

### 🟡 L2 — 过时内容

```
方法：
1. 检查每页 `updated` 日期
2. 超过30天未更新的 → 检查关键数据是否"最新"
3. 重点：股价/PE/目标价、行业格局、指标数值、报告结论
4. 对比 log.md 近期 ingest 记录中的新旧信息
```

| 页面 | 最后更新 | 过时内容 | 建议 |
|-----|---------|---------|------|
| stocks/X.md | 2026-03-01 | 股价是2月的 | 更新最新值 |
| themes/Y.md | 2026-02-15 | 政策预期旧 | 结合新ingest |

→ 能自动更新的直接更新 → 需要新资料的加入 reading-list

---

### 🟢 L3 — 孤岛页面

```
方法：
1. 列出 wiki/ 下所有 .md（排除 index/log/schema/_index-*）
2. 对每个非索引页，搜索全目录中被 [[名]] 或 (名.md) 引用的次数
3. 只有 _index-*.md 引用它而没有其他内容页链接 → 孤岛
```

| 页面 | 入链数(除索引) | 原因 | 建议 |
|-----|-------------|------|------|
| analyses/deep-dives/x.md | 0 | 创建后未关联 | 链接到相关个股/板块 |

→ 能修的直接加链接 → 真无关联的考虑保留必要性

---

### 🔵 L4 — 缺失实体

```
方法：
1. 搜索所有页面高频词汇（排除通用词）
2. 名词在 ≥3 页出现但没有独立实体/概念页
3. 且该名词是重要的可独立研究对象
→ 缺失实体
```

| 名称 | 提及次数 | 出现页面 | 建议类型 | 优先级 |
|-----|---------|---------|---------|-------|
| "CXL技术" | 5次 | stocks/澜起科技, sectors/AI算力,... | concepts/theories/CXL技术.md | 高 |

→ 高优先级 = 核心持仓相关 / 高频 / 有研究价值

---

### 🟣 L5 — 引用断裂

```
方法：
1. 正则提取所有 [[xxx]] 和 [text](path.md) 链接
2. 检查目标文件是否存在
3. 不存在 → 断裂
```

| 来源页面 | 链接文本 | 目标路径 | 是否存在 | 修复 |
|---------|---------|---------|---------|------|
| stocks/A.md | [[B公司]] | B公司.md | ❌ | 创建 or 改纯文本 |

→ 路径写错的直接修正 → 目标不存在的创建或去链

---

### 🟠 L6 — 信息空白

```
方法：
1. 审视主要实体页完整性
2. 对照模板检查缺失 section
3. 标记"应该有但目前没有"的信息维度
4. 判断能否通过 raw/ 已有资料 / 外部搜索填补
```

| 页面/实体 | 缺失信息 | 重要度 | 填补方式 | 建议 |
|----------|---------|-------|---------|------|
| stocks/澜起科技 | DDR5渗透率最新数据 | 高 | 需新研报 | 加reading-list |
| sectors/光伏 | 组件价格走势 | 中 | 可网络搜索 | 下次query补充 |

---

### ⚪ L7 — Schema 合规性

```
方法：
1. 抽查/全量检查每个 .md 文件
2. 检查项：
   - YAML frontmatter 完整性 (title/type/created/updated/sources/tags)
   - 命名是否符合约定
   - 目录是否正确
   - 日志前缀格式
   - 索引条目一致性
```

| 页面 | 问题 | 严重程度 | 修复 |
|-----|------|---------|------|
| xxx.md | 缺 tags 字段 | 低 | 补充 |
| yyy.md | 不在约定目录下 | 高 | 移动 |

## 自动修复 vs 人工确认

### ✅ 可自动修复（不问用户）
- 引用断裂（修正路径拼写错误）
- YAML frontmatter 缺少可选字段（补默认值）
- 孤岛页面添加合理入链
- 日志格式不规范（统一前缀）

### ⚠️ 必须标记待确认（不改原文件）
- 内容矛盾（需要判断哪个对）
- 删除任何实质性内容
- 修改 schema.md
- 合并或拆分页面

## 执行模式

### 完整模式（推荐每周一次）

全部7项 + 完整报告 + 自动修复 + Git commit

触发："对Wiki做Lint检查" / "全面检查知识库"

### 快速模式（随时可跑）

只跑 L1+L5+L2，不生成报告文件

触发："快速检查" / "有没有明显错误"

### 单项模式

只执行指定一项

触发："查一下有没有矛盾"(L1) / "看看孤岛页面"(L3)

## 注意事项

1. **Lint 是维护操作** — 不创建新的分析内容
2. **保守原则** — 宁可标记也不要误删有价值的内容
3. **报告必留档** — 每次 Lint 都存档追踪趋势
4. **不碰 raw/** — 原始资料不在范围
5. **不轻易改 schema.md** — 除非系统性问题
6. **频率建议**：日常使用每周1次 / 高频录入期每2-3天 / 低频使用每2周
